# Ergonomic Analysis Report

## Human-AI Decision Support for Satellite Ops Event Triage
### Simulation-Based Human Factors Evaluation

---

## 1. Context & Goal

Satellite operations require operators to triage continuous telemetry event streams, deciding which events to ignore, monitor, escalate, or act upon. AI decision support can accelerate this process, but introduces risks: automation bias, overconfidence in AI recommendations, and catastrophic errors when operators follow incorrect AI suggestions without verification.

This project evaluates three UI variants designed to mitigate these risks through a simulation-based human factors evaluation with computational operator models.

## 2. User Profiles & Scenarios

**Profiles:**
- **Novice** — High error rate (0.15), high automation reliance (0.85), low verification propensity
- **Experienced** — Low error rate (0.05), moderate reliance (0.55), higher verification propensity
- **Fatigued** — Rising error rate (0.10 + fatigue growth), declining attention and checklist adherence

**Scenarios:**
- **Routine shift** — 30 events, 10% high severity, low noise
- **Incident spike** — 60 events, 25% high severity, high noise, 20% interruption rate

## 3. Requirements Summary

Functional requirements focus on event display, AI recommendation visibility, evidence tracing, and telemetry recording. Ergonomic requirements target: context switching reduction, slip prevention, overload management, interruption recovery, and automation bias mitigation. See `docs/requirements.md` for full specification.

## 5. ML Pipeline

Following Turgeon & Lanovaz (2020), we use a transparent, reproducible ML workflow:

1. **Feature extraction:** 5 numeric features (latency, error count, anomaly score, trend slope, CPU usage) derived from noisy observations of a latent risk variable.
2. **Models:** Random Forest (primary) and Logistic Regression (comparator), both with balanced class weights.
3. **Evaluation:** 5-fold stratified cross-validation, confusion matrices, per-class recall (prioritizing run_procedure detection), Brier scores for calibration.
4. **Two-stage pipeline:** Severity classifier → policy mapping to action (cleaner ops story than direct action classification).

**Reference:** Turgeon, S., & Lanovaz, M. J. (2020). Tutorial: Applying machine learning in behavioral research. *Perspectives on Behavior Science, 43*(4), 697–723.

## 6. Simulation-Based Study Design

**Design:** Full factorial: 2 scenarios × 3 profiles × 3 UI variants = 18 conditions.
**Runs:** N = 1,000 simulated shifts per condition.
**Event streams:** Identical within each scenario across conditions (fair comparison).

**Operator decision model:** At each event, the computational operator:
1. Computes cognitive load from queue position and attention capacity
2. Decides whether to verify evidence (based on propensity, modified by UI)
3. Combines imperfect severity perception with AI recommendation
4. Applies checklist gate for high-risk actions (if applicable)
5. Outputs chosen action and sampled decision time

**Metrics collected:** Accuracy, miss rate, false alarm rate, catastrophic error rate, mean decision time, evidence view rate, checklist completion rate, appropriate/inappropriate reliance, operational cost.

## 7. Results

*Results are generated by running `python src/run_all.py`. See `outputs/tables/simulation_summary.csv` for full data and `outputs/figures/` for comparison plots.*

Key expected findings based on model mechanics:

- **Variant B** reduces inappropriate reliance on weak-evidence AI recommendations compared to A
- **Variant C** reduces catastrophic error rate through checklist gates
- **Variant C** reduces miss rate under incident spike through severity prioritization
- Decision time increases from A → B → C (tradeoff: safety vs speed)
- Fatigued operators benefit most from Variant C's cognitive supports
- Novice operators show largest reliance reduction from A → B

## 8. Ergonomic Findings

### Automation Bias
- Without evidence panels (Variant A), operators follow AI at rates matching their reliance parameter, regardless of evidence quality
- Evidence panels (B/C) create a verification pathway that reduces blind following, especially for weak-evidence events

### Alert Fatigue
- In incident scenarios (60 events), operators without prioritization (A/B) are more likely to miss high-severity events buried in the stream
- Variant C's priority sorting keeps critical events in the attention window

### Catastrophic Errors
- Checklist gates (Variant C) reduce catastrophic errors by forcing verification before high-risk actions
- However, checklist adherence decays with fatigue — fatigued operators skip checklists more often later in the shift

### Cognitive Load
- Decision time is a proxy for cognitive effort
- Variant C increases mean decision time but reduces error cost — a net positive in expected operational cost

## 9. Recommendations

### For Production Deployment
1. **Ship evidence panels** (Variant B minimum). The verification pathway reduces automation bias with acceptable time cost.
2. **Ship checklist gates** for high-risk actions (run_procedure). Despite time cost, the catastrophic error reduction justifies the friction.
3. **Consider adaptive prioritization** based on real-time event volume rather than static sorting.

### Guardrails
1. AI must report evidence strength. Never suppress uncertainty.
2. Checklist must not be bypassable without audit trail.
3. Monitor operator override patterns for trust calibration signals.

### Security Considerations
1. **Prompt injection risk** (conceptual): If an LLM generates explanations from log text, adversarial log entries could manipulate recommendations. Mitigation: use structured feature extraction, not raw text parsing.
2. **Model drift**: Monitor classifier performance against held-out validation data. Alert when accuracy drops below threshold.
3. **Audit trail**: All operator decisions + AI recommendations logged for post-incident review.

## 10. Limitations & Next Steps

### Limitations
- Synthetic operators, not real users — absolute metrics are not directly applicable
- Simplified cognitive model — real decision-making is more complex
- Feature space is deliberately limited — real telemetry has higher dimensionality
- See `docs/assumptions.md` for full validity limits

### Next Steps
1. **Wizard-of-Oz study** with 6–10 domain-adjacent participants
2. **Expert parameter review** by satellite ops SMEs
3. **Temporal modeling** — add circadian fatigue patterns and learning curves
4. **Real telemetry validation** — test DGP assumptions against mission logs
5. **Adaptive UI** — dynamically adjust interface features based on detected operator state
